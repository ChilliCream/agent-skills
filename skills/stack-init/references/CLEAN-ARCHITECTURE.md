# Clean Architecture reference

A four-layer pattern that isolates business rules from infrastructure and delivery mechanisms. Dependencies point inward only. The canonical .NET solution shape is `src/MyApp.{Domain,Application,Infrastructure,Web}/` — adapt the project names to whatever the target repo actually uses.

## Philosophy

Pick Clean Architecture when the domain and use cases need protection from delivery and infrastructure churn. It gives agents an obvious path for changes: Domain first for invariants, Application for use cases, Infrastructure for implementations, Presentation for transport wrappers.

## Layers

### Domain

The innermost layer. Pure business logic, no framework references, no `using Microsoft.EntityFrameworkCore;`.

Contains:

- **Entities** — classes with identity (`Guid Id`). Behaviour lives in methods, not setters. Constructors are `internal`, properties have `internal set` so only the layer itself and same-assembly factories can mutate state.
- **Value objects** — immutable records identified by their values, not identity (`AuthorName`, `Isbn`).
- **Domain events** — records that signal something happened (`BookRemovedEvent`). Stored on the entity's `Events` collection and dispatched later.
- **Invariants** — enforced in entity methods. A `Rename(string name)` method throws if `name` is empty; it does not silently accept bad input.

Forbidden: dependencies on Application, Infrastructure, or any framework other than the BCL.

### Application

Orchestrates use cases. Depends on Domain.

Contains:

- **Commands and command handlers** — one MediatR `IRequest` + handler per use case, colocated in the same file. Commands mutate state.
- **Queries** — read-only operations. A common convention is a class like `GetAuthorById` with an `ExecuteAsync` method that uses DataLoaders, not raw EF queries.
- **DataLoaders** — `[DataLoader]`-attributed static methods that batch reads by id.
- **DbContext abstraction** — an `IAppContext` interface that exposes `DbSet<T>`s. Application code uses the interface; Infrastructure provides the implementation. This keeps Application free of EF references in principle (in practice `DbSet<T>` lives in EF, but no migrations or providers do).
- **Domain event handlers** — react to domain events raised by entities (e.g., send an integration event when a `BookCreatedEvent` fires).

Validators are contested. Two camps:

- **Validators in Application** — `IValidator<CreateBookCommand>` runs as a MediatR pipeline behavior. Good when input shape validation is verbose and benefits from FluentValidation's API.
- **Invariants in Domain** — `Book.Create(title)` throws if `title` is empty. Good when the rule is intrinsic to the entity and would be wrong to bypass at any callsite.

Pick one and use it consistently. A common split is to keep invariants in Domain methods and reserve Application validators for shape checks (string length, regex).

### Infrastructure

Implementation details. Depends on Application.

Contains:

- **`IAppContext` implementation** — the concrete `AppContext : DbContext, IAppContext` class.
- **EF Core entity configurations** — `IEntityTypeConfiguration<T>` classes inside `EntityConfigurations/`.
- **Migrations** — generated by `dotnet ef migrations add`.
- **External service clients** — HTTP, message bus, blob storage. These live behind interfaces declared in Application.

### Presentation (GraphQL / REST / CLI)

The thin delivery layer. Depends on Application.

For GraphQL services, this is `*.GraphQL` (or `*.Web`). Conventions:

- `[ObjectType<T>]` static partial classes mirror domain entities. The source generator builds the schema.
- Resolvers delegate to Application: a query resolver calls `GetAuthorById.ExecuteAsync`, a mutation resolver sends a `CreateBookCommand` via Mediator.
- No business logic. If a resolver would need an `if`, it belongs in the Application layer.

REST controllers and CLI commands follow the same rule — they translate transport-specific input into commands and queries, nothing more.

## Dependency rules

```
Presentation ──► Application ──► Domain ◄── Infrastructure
```

- Domain depends on nothing.
- Application depends on Domain.
- Infrastructure depends on Application (it implements the abstractions Application declares).
- Presentation depends on Application.
- Presentation does **not** reference Infrastructure directly — DI wires the implementations at composition root (typically the `*.Host` project).

Enforce this with `.csproj` references. If the dependency graph compiles, the rules hold.

## Folder layout (typical .NET solution)

```
src/
  MyApp.Domain/
    Entities/
      Authors/
        Author.cs
        Events/
          AuthorCreatedEvent.cs
      Books/
        Book.cs
        Events/
          BookAddedEvent.cs
    Common/
      Entity.cs
      AuthorName.cs

  MyApp.Application/
    Entities/
      Authors/
        CreateAuthorCommand.cs          # command + handler in one file
        GetAuthorById.cs                # query class
        AuthorDataLoaders.cs
      Books/
        AddBookCommand.cs
    Common/
      IAppContext.cs

  MyApp.Infrastructure/
    AppContext.cs
    EntityConfigurations/
    Migrations/

  MyApp.Web/
    Types/
      AuthorType.cs
      BookType.cs
    Mutations/
```

The same pattern shape works for any service-shaped repo. Replace `MyApp` with the target repo's actual project prefix.

## Worked example — adding a new entity

Adding a `Book` entity end-to-end:

1. **Domain.** Create `Domain/Entities/Books/Book.cs`:

   ```csharp
   public sealed class Book : Entity
   {
       internal Book(Guid id, Guid authorId, string title, DateTimeOffset createdAt)
       {
           Id = id;
           AuthorId = authorId;
           Title = title;
           CreatedAt = createdAt;
       }

       public Guid Id { get; internal set; }
       public Guid AuthorId { get; internal set; }
       public string Title { get; internal set; }
       public DateTimeOffset CreatedAt { get; internal set; }
       public bool IsPublished { get; internal set; }

       public static Book Create(Guid authorId, string title)
       {
           if (string.IsNullOrWhiteSpace(title))
               throw new ArgumentException("Title cannot be empty.", nameof(title));

           var b = new Book(Guid.CreateVersion7(), authorId, title, DateTimeOffset.UtcNow);
           b.Events.Add(new BookCreatedEvent(b));
           return b;
       }

       public void Publish()
       {
           if (IsPublished) return;
           IsPublished = true;
           Events.Add(new BookPublishedEvent(this));
       }
   }
   ```

2. **Application.** Add `Application/Entities/Books/CreateBookCommand.cs` with the MediatR command and handler colocated. The handler calls `Book.Create(...)` and adds it to `IAppContext.Books`. Add `BookDataLoaders.cs` with `GetBookByIdAsync`.

3. **Infrastructure.** Add `DbSet<Book> Books` to `IAppContext` (interface in Application) and to `AppContext` (implementation in Infrastructure). Add `BookConfiguration : IEntityTypeConfiguration<Book>`. Run `dotnet ef migrations add AddBooks`.

4. **Web.** Add `Web/Types/BookType.cs`:

   ```csharp
   [ObjectType<Book>]
   public static partial class BookType
   {
       [NodeResolver]
       public static Task<Book?> GetBookAsync(
           Guid id,
           ISession session,
           GetBookById service,
           CancellationToken ct)
           => service.ExecuteAsync(session, id, ct);
   }
   ```

   Add a `CreateBook` mutation (or REST endpoint) that constructs the command and sends it via Mediator.

Each step touches one layer only. If you find yourself reaching across layers, the abstraction is wrong.

## Agentic coding preparation

`ARCHITECTURE.md` should make these rules explicit:

- Domain has no framework dependencies and owns invariants.
- Application exposes commands, queries, DataLoaders, authorization, and abstractions.
- Infrastructure implements Application abstractions and owns provider details.
- Presentation/GraphQL is thin and delegates to Application.
- Every feature plan starts by naming the layer touched by each step.

`DOMAIN.md` should name aggregate roots, child entities, value objects, domain events, and invariants so an agent does not invent duplicate concepts.
